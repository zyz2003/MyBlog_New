  <name>Task 1: Auth composable + Pinia store + route middleware</name>
  <files>apps/site/composables/useAuth.ts, apps/site/stores/auth.ts, apps/site/middleware/admin-auth.ts</files>
  <behavior>
    - useAuth composable exposes: login(username, password), logout(), refreshToken(), user (ref), token (ref), isAuthenticated (computed)
    - login() calls POST /api/auth/login, stores token in localStorage, updates Pinia store
    - logout() calls POST /api/auth/logout, clears token from localStorage, clears store, navigates to /admin/login
    - refreshToken() calls POST /api/auth/refresh with current token, updates stored token
    - On composable init (client-side): read token from localStorage, validate via /api/auth/refresh if needed, fetch user info
    - Pinia auth store: state = { user, token }, getters = { isAuthenticated }
    - admin-auth middleware: runs on client-side route navigation, checks isAuthenticated, redirects to /admin/login if not authenticated
    - Middleware skips /admin/login route itself
  </behavior>
  <action>
Create three files:

1. `apps/site/stores/auth.ts` — Pinia store:
   - State: `user: { id, username, email, displayName, avatar, role } | null`, `token: string | null`
   - Getter: `isAuthenticated` — true when token is not null
   - Actions: `setAuth(token, user)`, `clearAuth()`, `setToken(token)`
   - Use `defineStore('auth', ...)` with setup syntax

2. `apps/site/composables/useAuth.ts` — composable wrapping the store:
   - `login(username: string, password: string)`: POST to `/api/auth/login` with `{ username, password }`. On success: call `store.setAuth(data.token, data.user)`, save token to `localStorage.setItem('auth_token', token)`. On error: throw the error message.
   - `logout()`: POST to `/api/auth/logout` (best-effort, don't fail if error). Call `store.clearAuth()`, `localStorage.removeItem('auth_token')`, `navigateTo('/admin/login')`.
   - `refreshToken()`: POST to `/api/auth/refresh` with `Authorization: Bearer {token}` header. On success: update token in store and localStorage. On 401: call logout().
   - `initAuth()`: On client-side only. Read token from localStorage. If exists: set in store, try refreshToken(). If refresh fails (401), clearAuth. If refresh succeeds, the token is fresh.
   - Export `useAuth()` function returning `{ login, logout, isAuthenticated, user, token }`.
   - Call `initAuth()` automatically on first `useAuth()` invocation (use a flag to run once).

3. `apps/site/middleware/admin-auth.ts` — client-side route middleware:
   - Use `defineNuxtRouteMiddleware`
   - Skip if route path is `/admin/login`
   - Import auth store, check `isAuthenticated`
   - If not authenticated, `return navigateTo('/admin/login')`
   - If authenticated, allow navigation

   Important: This middleware is registered globally via `middleware/admin-auth.ts` file name convention. Nuxt auto-registers files in `middleware/` directory. To apply it only to admin routes, use `definePageMeta({ middleware: 'admin-auth' })` in admin pages, OR use the `pages/admin.vue` layout approach.
   Best approach: Create `pages/admin.vue` as a wrapper that applies the middleware via `definePageMeta`.

| Boundary | Description |
|----------|-------------|
| client→API | Login form sends credentials to /api/auth/login |
| localStorage→state | Token stored in localStorage, read on page load |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | I | localStorage token | accept | JWT has 7-day expiry, auto-refresh mitigates long-lived token risk. SPA requires client-side storage. |
| T-07-02 | S | Login credentials | mitigate | POST /api/auth/login already has rate limiting (5/min per API-12). HTTPS required in production. |
| T-07-03 | T | Token interception | accept | JWT is signed (HS256), cannot be tampered. HTTPS in production protects transport. |
| T-07-04 | E | Auth bypass | mitigate | admin-auth middleware checks store state AND validates token via refresh on init. |
</threat_model>
